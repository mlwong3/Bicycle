# 騎跡管理員系統 Phase 1 設計規格

> 日期：2026-07-13  
> 狀態：已獲使用者批准的設計方向，待文件審閱後進入實作

## 1. 目標與範圍

本階段只完成管理員端的基礎閉環，讓一宗市民舉報可以在同一個示範系統內被查看、接收、更新狀態及追蹤處理歷史。

本階段同時把「停泊時長推斷」改為「疑似棄置風險評估」，但不把真實 AI 接入正式案件流程。AI 只保留為獨立試運行項目，試運行結果不會改變正式 `Report` 狀態，也不會直接影響正式巡邏路線。

## 2. 明確不包括的內容

- 不在本階段正式加入 Gemini Vision。
- 不把單張相片視為可證明停泊時間的證據。
- 不在本階段實作巡邏路線演算法；只準備座標及案件狀態資料。
- 不把示範密碼當作正式政府身份驗證。
- 不為了展示而放寬 Firestore 規則至所有匿名使用者可讀寫全部案件。

## 3. Phase 1 功能設計

### 3.1 資料模型

`Report` 新增或擴充：

- `lat`、`lng`：GPS 成功取得時保存；沒有座標的手動位置仍可提交，但不能納入路線計算。
- `status`：`pending`、`reviewing`、`noticed`、`scheduled`、`resolved`、`dismissed`。
- `statusHistory[]`：每次狀態變更追加時間、操作者及備註。
- `noticeDate`：示範流程用日期，不宣稱適用於所有現實個案。
- `handledBy`：示範管理員識別資料。

AI專用欄位暫不成為 Phase 1 正式資料流程；試運行資料另行保存，避免未驗證 AI結果污染案件紀錄。

### 3.2 管理員示範模式

- MenuSidebar 新增「管理員模式」。
- 使用 `VITE_ADMIN_DEMO_PASSWORD`，沒有設定時使用明確標示的示範 fallback。
- 通過後以 `sessionStorage` 保存示範 session。
- 管理員畫面顯示「示範模式」及「正式系統需使用政府帳戶及多重驗證」。
- 管理員狀態寫入 Firebase 時採 best-effort；若權限或網絡不允許，仍可使用本機示範資料完成展示。

### 3.3 AdminTab

AdminTab 接收 App 現有的 reports 狀態及更新回呼，提供：

1. 案件列表：按狀態顯示案件卡片、位置、日期及圖片。
2. 案件詳情：描述、座標、示範標籤及目前狀態。
3. 狀態轉換：依照狀態機更新案件，並追加 `statusHistory`。
4. 時間線：按時間顯示處理歷史。
5. 示範資料重設：恢復預載案件，方便現場重播 Demo。

市民端只把新增狀態映射為「處理中」、「已清理」或「不成立」，避免 Phase 1 大幅改寫原有 UI。

### 3.4 舉報座標及圖片

- ReportTab 使用現有 GPS 取得流程，成功時把 `lat/lng` 傳回 App。
- 圖片上傳採「Firebase Storage 可用時使用，失敗時保留本機 data URL」的 fallback。
- 沒有座標的案件仍可在管理員端查看及更新，但 AdminTab 不應把它加入路線資料。

## 4. AI試運行設計

AI試運行不接入正式 AdminTab，使用獨立測試輸入及輸出。

### 4.1 名稱及輸出

所有文件及未來介面使用「疑似棄置風險評估」，不再使用「停泊時長推斷」。建議輸出：

```json
{
  "isBicycle": true,
  "riskScore": 62,
  "riskLevel": "中高風險",
  "confidence": "medium",
  "indicators": {
    "rust": 2,
    "tire": 3,
    "dust": 2,
    "attachment": 1,
    "missing": 0,
    "lock": 1
  },
  "reasons": ["鏈條及輪圈有明顯鏽蝕", "前輪胎完全扁塌"],
  "suggestedAction": "建議管理員現場覆核"
}
```

### 4.2 試運行判斷標準

- 能否辨認相片是否為單車。
- 相片特徵與風險評級是否大致合理。
- `confidence` 是否能反映圖片模糊、局部或資訊不足。
- 建議行動是否始終保留人員覆核。
- 結果是否能清楚說明「不是確實停泊時間」。

試運行完成後，才決定是否建立正式 `ai.ts`、是否保存 `aiAnalysis`，以及是否讓 AI結果參與案件排序。

## 5. AI巡邏路線的可行性及實用性

Phase 1 只準備資料，不實作路線。日後 Phase 3 應採「示範優化路線」而不是未經驗證的「最優清理路線」。

### 5.1 可行部分

- 使用現有 GPS、Leaflet、Mapbox 及 `trackRouting.ts`。
- 對 5–10 宗有座標案件使用 Haversine 距離、最近鄰居法及 2-opt，技術上足以支援比賽 Demo。
- 先比較舉報次序與優化次序，再顯示距離節省百分比。
- 無法連接官方單車徑時，依次退回道路路線及直線估算，並標示路線來源。

### 5.2 實用性限制

路線排序不等於真實執法決策，日後需考慮：案件優先級、工作人員每日容量、限期、回收站位置、道路及單車徑連通性、封路、天氣、現場安全及跨部門安排。

因此日後驗收不只看距離，還要記錄：

- 距離節省百分比。
- 高風險案件覆蓋率。
- 預計處理時間。
- 路線失敗及 fallback 次數。
- 沒有座標或無法實際到達的案件數量。

## 6. 資料流

```text
市民 ReportTab
  -> Report { location, lat, lng, imageUrl, status: pending }
  -> App state + localStorage
  -> Firebase best-effort sync
  -> 管理員示範登入
  -> AdminTab 查看及更新狀態
  -> statusHistory 追加紀錄
  -> 市民端映射顯示處理中／已清理
```

AI試運行另行處理：

```text
測試圖片 -> AI試運行 -> riskScore/riskLevel/confidence -> 測試紀錄
                                      X 不寫入正式 Report
                                      X 不觸發正式巡邏路線
```

## 7. 驗收準則

- `Report` 可保存座標及新的狀態值。
- 管理員示範登入及登出可正常運作，重新整理頁面後 session 行為一致。
- 管理員可以查看案件、更新狀態及看到完整狀態時間線。
- 市民端不因新增狀態而出現 TypeScript 錯誤或錯誤文案。
- 沒有 Firebase 或 Firebase 權限失敗時，示範仍可用本機資料完成。
- 沒有座標的案件不會被納入日後路線資料。
- AI試運行結果不會寫入正式 `Report` 或改變案件狀態。
- 完成型別檢查及 Vite build；在含 `#` 的 iCloud路徑不能建置時，改於不含 `#` 的暫存路徑驗證。
