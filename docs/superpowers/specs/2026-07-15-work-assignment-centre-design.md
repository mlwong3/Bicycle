# Bike Trace 工作分配中心設計規格

日期：2026-07-15  
狀態：Phase 1 設計已獲使用者確認，待規格檢視後制定實作計劃

## 1. 設計目的

現有案件只有 `assignedDepartment`、`assignedTeam` 及 `handledBy` 等單一指派欄位，未能表達香港清理違例或疑似棄置單車時，一宗案件可能涉及多個部門、不同法定步驟、先後依賴及聯合行動日期。

本設計新增「工作分配中心」，把案件轉化為多張可追蹤的工作單，支援：

- 按地點、管理權責及案件情況建議主責和協作部門；
- 由統籌人員確認分流，不由 AI 自動作出執法決定；
- 把一宗案件拆成多項有先後次序的部門工作；
- 記錄接收、排期、執行、受阻、完成及重新指派；
- 只把已排期、同類型及同部門的工作交給巡邏路線規劃；
- 保留完整審計紀錄，供比賽示範及日後真實部署評估。

系統定位維持為「跨部門單車執法決策支援及聯合行動管理原型」，不聲稱取代政府程序或擁有執法權。

## 2. 核心原則

### 2.1 一宗案件，多張工作單

案件代表整體事件；工作單代表某部門、團隊或人員要完成的一項具體工作。部門完成自己的工作單，不等於整宗案件已結案。

### 2.2 規則輔助，人員確認

系統可依據已確認的處理路徑建立工作單及推薦合資格團隊，但統籌人員必須確認指派。AI 只可摘要案件、提示資料不足或提出非約束性建議。

### 2.3 法定權責是硬性條件

部門權責、土地類別、設施管理權及程序先後不得以「評分較高」取代。未確認管轄權時，案件須進入人工覆核，不得自動派發清理工作。

### 2.4 案件狀態與工作狀態分離

- 案件狀態：反映整體行政或程序階段。
- 工作單狀態：反映個別部門工作的執行情況。
- 聯合行動狀態：反映指定地點和日期的跨部門準備程度。

## 3. 三層工作分配

### 3.1 部門分流

系統先按人工確認的地點類別和處理路徑，建立主責或協作部門工作單。

| 情況 | 建議分流方式 |
|---|---|
| 即時危險或嚴重阻塞 | 警務處優先處理；相關管理部門提供支援 |
| 公眾地方殘破並被確認為街道廢物 | 食環署核實及移走 |
| 未批租政府土地 | 地政總署負責法定通知及接管；按程序建立移走支援工作 |
| 公共單車泊車處聯合行動 | 民政處統籌；運輸署、警務處、地政總署及食環署按程序分工 |
| 運輸署管理設施 | 運輸署主責，按需要建立其他部門支援工作 |
| 管轄權未明 | 只建立「管轄權覆核」工作，不建立清理或充公工作 |

部門名稱和流程模板均須可配置，避免把原型的示範資料寫成不可更改的官方規則。

### 3.2 團隊分配

部門確定後，系統按地區、職能、所需設備、排期及現時工作量，推薦合資格團隊。統籌人員或部門主管確認後，團隊可接收或附理由退回。

### 3.3 個人執行

團隊主管把已接收工作指派給個別人員。前線人員只看到其所屬部門和獲授權地區內的工作，並可執行「開始、受阻、完成、上載證據」等操作。

## 4. 資料模型

### 4.1 WorkOrder 工作單

建議欄位：

| 欄位 | 用途 |
|---|---|
| `id` | 工作單識別碼 |
| `caseId` | 所屬案件 |
| `jointOperationId` | 所屬聯合行動，可留空 |
| `taskType` | 工作類型，如覆核、通知、封閉、移走、接管、結案覆核 |
| `leadDepartment` | 執行此工作單的主責部門 |
| `supportingDepartments` | 需要支援的部門 |
| `assignedTeamId` | 獲指派團隊 |
| `assignedStaffUid` | 獲指派人員 |
| `district` | 執行地區 |
| `scheduledAt` | 預定開始時間 |
| `dueAt` | 截止時間；由程序配置或人員設定 |
| `priority` | 緊急程度 |
| `prerequisiteWorkOrderIds` | 必須先完成的工作單 |
| `requiredCapabilities` | 所需職能或授權 |
| `requiredEquipment` | 所需車輛或設備 |
| `evidenceChecklist` | 完成前必須提交的證據 |
| `status` | 工作單狀態 |
| `blockerReason` | 受阻原因 |
| `assignmentHistory` | 接收、退回及重新指派紀錄 |
| `createdAt` / `updatedAt` | 時間紀錄 |

### 4.2 JointOperation 聯合行動

建議欄位包括：行動地點、行動日期、統籌部門、參與部門、必須完成的工作單、準備狀態、路線分組及結案狀態。

### 4.3 Team 團隊

建議欄位包括：所屬部門、服務地區、可執行工作類型、設備、當值狀態、每日容量及現時工作量。

## 5. 工作單狀態

正常流程：

`draft → awaiting_acceptance → accepted → scheduled → in_progress → completed`

例外狀態：

- `blocked`：因程序、證據、天氣、設備或其他前置條件受阻；
- `declined`：團隊附理由退回統籌人員；
- `cancelled`：工作不再需要，但保留紀錄。

狀態規則：

1. 前置工作未完成時，不得開始下游工作。
2. 法定或程序限期未屆滿時，不得把移走工作標記為可執行。
3. 完成工作前，必須符合該工作類型的證據清單。
4. 重新指派必須填寫理由並保留前後負責者。
5. 所有必須工作單完成後，案件才可結案。

## 6. 規則輔助分配流程

1. 統籌人員確認地點類別、管理權責及處理路徑。
2. 系統按可配置程序模板，把案件展開成一組工作單。
3. 系統以部門權責、地區、團隊職能及設備作硬性篩選。
4. 系統檢查工作依賴及可執行日期。
5. 系統在合資格團隊中計算推薦次序。
6. 統籌人員確認或修改分配。
7. 團隊接收、排期及指派人員。
8. 人員執行並提交證據；受阻時通知統籌人員。
9. 系統在所有必要工作完成後，容許案件結案。

建議的團隊推薦權重：

| 因素 | 權重 |
|---|---:|
| 緊急程度 | 35% |
| 截止時間接近程度 | 25% |
| 地區配對 | 15% |
| 職能及設備配對 | 15% |
| 現時工作量 | 10% |

以上權重只用於已通過權責和資格篩選的團隊。即時危險案件須繞過一般輪候次序，直接進入緊急分流。

## 7. 公共單車泊車處聯合行動示例

| 次序 | 工作單 | 執行部門 | 前置條件 |
|---:|---|---|---|
| 1 | 發出暫停使用泊車處通知 | 運輸署 | 無 |
| 2 | 封閉泊車處或處理告示 | 警務處 | 工作 1 完成 |
| 3 | 在單車張貼法定通知 | 地政總署 | 工作 2 完成 |
| 4 | 行動日移走單車 | 食環署 | 工作 2、3 完成及相關期限屆滿 |
| 5 | 接管、保管及後續處置 | 地政總署 | 工作 4 完成 |
| 6 | 聯合行動覆核及結案 | 民政處 | 所有必要工作完成 |

此示例用作原型流程模板。真實部署前須由相關部門核實法定程序、時限、職能名稱及資料交換要求。

## 8. 使用者介面

### 8.1 工作分配中心

後台新增以下視圖：

- 未分配工作：顯示待確認部門或團隊的工作單；
- 部門工作板：按「待接收、已排期、進行中、受阻、已完成」排列；
- 聯合行動日曆：顯示行動日期、參與部門及準備程度；
- 團隊容量：顯示各團隊當值、設備及工作量；
- 案件工作樹：顯示一宗案件的所有工作及先後依賴；
- 我的工作：供前線人員開始、受阻、完成及提交證據。

### 8.2 權限

- 統籌人員：建立聯合行動、確認部門分流、處理退回及重新指派；
- 部門主管：接受工作、排期、選擇團隊及人員；
- 前線人員：更新獲指派工作及提交證據；
- 系統管理員：管理模板、部門、團隊和示範帳戶；
- 評審示範帳戶：只讀或使用獨立示範資料。

### 8.3 跨部門統籌儀表板

Phase 1 同時提供一個精簡的跨部門統籌儀表板，主要供民政處或聯合行動統籌人員掌握整體進度。儀表板不作正式部門績效評核，也不使用 AI 自動批准工作。

儀表板包括：

1. **概況卡**：待分流案件、即時危險案件、待部門接收工作、已排期工作、受阻或逾期工作、今日聯合行動。
2. **優先處理清單**：依次顯示即時危險、受阻、即將到期、等待接收及管轄權未明的工作，並提供查看案件和進行分配入口。
3. **跨部門準備進度**：按聯合行動顯示各部門工作、前置條件及目前狀態，讓統籌人員辨識尚未完成的環節。
4. **部門工作量概況**：顯示各部門待接收、進行中、受阻及今日完成數量，只作資源協調提示。
5. **今日行動及路線**：按現場核實、張貼通知、移走及運送分類，顯示已排期且符合執行條件的工作。

儀表板資料全部由案件、工作單、聯合行動及巡邏路線資料即時計算，不另設一套互相重複的統計資料。使用者可按地區、部門、日期及狀態篩選，並由數字或清單直接進入相應案件或工作單。

Phase 1 儀表板暫時協助政府人員：

- 快速識別最緊急、受阻及即將到期的案件；
- 掌握跨部門工作交接，減少工作被遺漏；
- 防止前置程序未完成便錯誤安排下游工作；
- 在聯合行動前檢查各參與部門是否準備妥當；
- 集中安排同區及同類型的外勤工作；
- 以單一畫面準備跨部門會議或行動簡報。

所有畫面須標示為「原型示範資料」，避免評審或試用者誤以為正在顯示真實政府案件或正式部門績效。

## 9. 與 AI 巡邏路線規劃的整合

巡邏路線不應直接使用所有案件，而應只讀取已排期且可執行的工作單。系統須分開以下路線：

- 現場核實路線；
- 張貼通知路線；
- 移走及運送路線。

只有同一部門、同一行動日期、相容工作類型及相容設備的工作才可放入同一路線。涉及不同法定步驟、未完成前置工作或限期未屆滿的工作不得混入。

路線模組的實用價值在於減少重複往返、按地區集中工作及協助主管排班；它不決定應否執法，也不更改部門權責。原型階段可使用確定性排序演算法，並把「AI 路線建議」標示為可由人員調整的建議。

## 10. 例外及錯誤處理

- 團隊退回：必須選擇原因，工作返回統籌佇列。
- 工作受阻：凍結所有依賴該工作的下游工作。
- 無合資格團隊：不作自動指派，改由統籌人員處理。
- 管轄權不明：只建立覆核工作，不建立清理工作。
- 重複案件：只取消尚未開始的重複工作單，保留歷史紀錄。
- 證據不足：不得完成工作，顯示尚欠項目。
- 限期未屆滿：移走工作維持鎖定。
- 聯合行動延期：重新計算相關排期，但不刪除原來審計紀錄。

## 11. Phase 1 唯一實作範圍

- 建立工作單、聯合行動及團隊資料模型；
- 設置五個示範部門，每個部門兩個示範團隊；
- 提供三個流程模板：即時危險、街道廢物、公共泊車處聯合行動；
- 支援確認分配、接收、排期、開始、受阻、完成及重新指派；
- 支援工作依賴、證據清單及審計紀錄；
- 建立供跨部門統籌人員使用的精簡儀表板；
- 支援地區、部門、日期及狀態篩選；
- 巡邏規劃改為讀取已排期工作單；
- 使用示範帳戶及本地／測試資料，不接駁真實政府系統；
- 提供示範數據及自動測試，支援完整比賽演示流程。

本次不設 Phase 2 或 Phase 3 實作。AI 自動批准工作分配、真實政府帳戶、真實執法通知、正式部門績效評核、長期趨勢預測及未經核實的官方系統整合均不納入範圍。

## 12. 驗收準則

1. 一宗公共泊車處示範案件可產生六張有依賴關係的工作單。
2. 不具權責的部門不會出現在候選名單。
3. 前置工作未完成時，下游工作不能開始。
4. 緊急案件可繞過一般輪候流程。
5. 相同輸入產生相同團隊推薦次序，並可解釋評分原因。
6. 重新指派前後的負責團隊、時間及原因均獲保留。
7. 所有必要工作未完成前，案件不能結案。
8. 巡邏路線只包含已排期、可執行、同部門及相容類型的工作。
9. 無合資格團隊或管轄權未明時，系統安全地轉交人工處理。
10. 評審可在一次示範中看到「案件建立 → 多部門派工 → 依次完成 → 路線建議 → 結案」。
11. 儀表板概況卡、優先清單及部門工作量均由同一組案件和工作單資料計算。
12. 變更地區、部門、日期或狀態篩選後，所有儀表板區塊顯示一致結果。
13. 未完成前置工作的下游工作不會出現在今日可執行路線。
14. 儀表板清楚標示為原型示範資料，不會把數量描述成真實政府績效。

## 13. 科創作品展示價值

本功能把作品由單純「把案件轉交某部門」提升為可解釋的跨部門程序原型。核心展示句為：

> 一宗案件、多張工作單；每項必要工作完成後，案件才推進至下一個程序。

評審可清楚看到系統如何運用規則、資料結構、權限、演算法及人機協作解決實際協調問題，同時避免把 AI 誤作執法決策者。

## 14. 待真實部署確認事項

- 各地點類別的正式部門權責及內部轉介安排；
- 各通知、限期、移走及處置程序的法定依據和最新版本；
- 路政署在單車個案中的適用情況，不預設為一般單車流程；
- 部門、團隊、職級、設備和地區的真實資料；
- 完成各工作所需證據及保存年期；
- 跨部門資料共享、私隱、保安及審計要求。
