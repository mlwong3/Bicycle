# 《騎跡 Bike Trace》技術深化簡報設計規格

## 1. 溝通任務

在約 8 分鐘內，讓 STEAM 比賽評審相信《騎跡》並非只有介面概念，而是一個把香港單車開放數據、真實定位與路線規劃、NFC、ESP32 感測器、Firebase 雲端資料及泊位預測整合起來的可展示原型；同時誠實區分已完成、示範模式及未來發展。

## 2. 對象及預期反應

- 對象：STEAM／創新科技比賽評審。
- 已知程度：理解一般 App、IoT 和雲端概念，但未必熟悉 React、Realtime Database 或路線規劃細節。
- 預期反應：評審能指出作品的技術核心、看見軟硬件資料確實流動，並相信學生理解各模組如何合作。

## 3. 核心訊息

《騎跡》的技術價值不在於單一功能，而在於把「政府開放數據、使用者手機、雲端資料庫及實體感測器」連成一個具備離線備援的單車管理流程。

## 4. 敘事路線

採用「問題 → 系統回答 → 技術拆解 → 現場證據 → 價值與限制」的累進結構。先用兩頁建立真實需要，之後以一頁總架構作導航，再按地圖、IoT、預測、NFC、舉報、雲端及備援逐層解釋，最後用現場示範和限制說明收結。

## 5. 投影片規格

### 第 1 頁：騎跡——讓每一次騎行都有數據可循

- 任務：以一句話定位作品。
- 畫面：沿用舊版淺綠底、青綠橫帶、深藍字及單車主視覺。
- 講稿時間：約 20 秒。

### 第 2 頁：單車使用者面對的不是一個問題，而是一條斷裂的流程

- 任務：歸納泊位難找、失竊／身份識別、違規及回收資訊分散三個痛點。
- 內容只使用原有項目資料；沒有來源支持的統計數字不寫成已證實成果。
- 講稿時間：約 35 秒。

### 第 3 頁：我們把四個資料來源連成一個單車管理系統

- 任務：用總架構圖交代使用者、React App、政府開放數據、Firebase、ESP32／NFC 的關係。
- 資料流：使用者操作 → React App → CSDI／Mapbox／Firebase；ESP32 → RTDB → 地圖；localStorage 作本機備援。
- 講稿時間：約 40 秒。

### 第 4 頁：App.tsx 是總控制室，各功能模組各自負責一件事

- 任務：顯示程式模組分工，而非展示大量原始碼。
- 模組：`App.tsx`、`MapTab.tsx`、`NfcTab.tsx`、`ReportTab.tsx`、`PersonalTab.tsx`、`backend.ts`。
- 講稿時間：約 35 秒。

### 第 5 頁：地圖不只顯示位置，而是組合四層真實資料

- 任務：說明 Leaflet 底圖、運輸署 CSDI 單車泊位／單車徑、地政總署 WMS 圖層、GPS 使用者位置。
- 清楚指出運輸署泊位數據是登記容量，不等同即時空位；即時空位來自 ESP32。
- 講稿時間：約 40 秒。

### 第 6 頁：路線規劃會按可用資料逐級降級

- 任務：解釋單車徑優先路線、Mapbox Cycling、直線示範三層 fallback。
- 加入地點搜尋、反向地址、GPS 導航及估算距離／時間。
- 講稿時間：約 40 秒。

### 第 7 頁：ESP32 把實體泊位狀態變成可讀數據

- 任務：展示 HC-SR04 量距 → ESP32 判斷佔用 → Wi-Fi → Firebase RTDB。
- 安全重點：Echo 由 5V 分壓至約 3.3V；簡報不顯示 Wi-Fi 密碼或 API 憑證。
- 講稿時間：約 40 秒。

### 第 8 頁：網站可即時知道每個泊位裝置是否在線

- 任務：解釋 `/parking/{deviceId}` 的 `total`、`free`、位置及 `updatedAt`，以及裝置過期判斷。
- 畫面：地圖上的 IoT 標記、即時空位及最後更新時間。
- 講稿時間：約 35 秒。

### 第 9 頁：預測模型用歷史時段回答「一小時後還有位嗎？」

- 任務：以中學生能解釋的方式介紹 `parkingHistory` 和 `predict.ts`。
- 不把簡單統計預測包裝成高階 AI；說明樣本不足時回傳無預測結果。
- 講稿時間：約 35 秒。

### 第 10 頁：NFC 只存識別碼，個人資料留在受控資料層

- 任務：交代 Web NFC 真實讀取、文字／JSON 相容、QR／模擬 fallback 及最少資料原則。
- 清楚區分 Android Chrome 真 NFC 與其他裝置的替代流程。
- 講稿時間：約 35 秒。

### 第 11 頁：舉報資料由手機產生，再同步至雲端

- 任務：展示相片、位置、描述、狀態和積分的資料流。
- 政府部門跟進只標示為模擬工作流程，不聲稱是真實法定程序。
- 講稿時間：約 30 秒。

### 第 12 頁：Firestore 和 RTDB 各自處理不同類型的資料

- 任務：比較 Firestore 儲存單車、NFC、舉報、行程；RTDB 處理快速變動的泊位狀態及歷史。
- 補充匿名登入、擁有者權限及環境變數設定。
- 講稿時間：約 40 秒。

### 第 13 頁：即使網絡或服務失效，核心展示仍可繼續

- 任務：說明 localStorage、種子資料、地圖資料 fallback、導航示範模式及預錄 Demo。
- 把可靠性設計轉化為評審可理解的工程亮點。
- 講稿時間：約 35 秒。

### 第 14 頁：現場用一輛模型單車證明整條資料鏈

- 任務：用 45 至 60 秒示範收結：放入模型單車 → 感測器讀值 → RTDB 更新 → 地圖空位改變 → 顯示預測／NFC 登記。
- 結語：作品目前是可展示原型；下一步是真實場地測試、收集更多歷史數據及加入管理員角色。
- 講稿時間：約 50 秒。

總口述時間目標：約 7 分 50 秒，保留約 10 秒處理頁面切換或短暫停頓。

## 6. 視覺方向

- 保留舊簡報辨識度：淺薄荷綠背景、青綠色標題帶、深藍標題、綠色重點。
- 版面由舊版的多個細小文字框，改為一頁一個技術主張。
- 以現有 App 畫面、程式碼截圖、實物／感測器照片和少量原生流程圖為主要視覺。
- 程式碼只截取 4 至 8 行關鍵片段；不放整頁 IDE。
- 圖示只作輔助，不使用與功能無關的裝飾圖片。
- 所有密碼、API 金鑰、Firebase 憑證和可識別個人資料必須遮蔽或排除。

## 7. 講稿文件

- 另建可編輯 Word 文件，每頁包括：投影片標題、建議時間、逐字講稿、畫面／Demo 操作提示、可能被追問的技術補充。
- 語氣採用香港中學生可自然說出的正式口語，不用艱深術語堆砌。
- 每段先講功能價值，再解釋技術，最後指出限制或證據。

## 8. 內容邊界

- 「已完成」必須可從現有程式碼或可運行介面核對。
- 「模擬」包括沒有真實 GPS 時的導航播放、政府跟進流程及尚未取得真實場地數據的部分。
- 「未來發展」不混入現時成果頁。
- 泊位預測稱為「簡單統計預測模型」，除非程式實際加入機器學習模型，否則不稱為 AI 預測。
- 不聲稱已完成大規模使用者測試或政府合作。
